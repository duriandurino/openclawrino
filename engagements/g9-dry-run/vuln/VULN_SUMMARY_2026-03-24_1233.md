## 4. Exploitability Decision Criteria

For each potential vulnerability, the following criteria determine whether exploitation proceeds:

| Criterion | Questions Asked | Threshold |
|-----------|-----------------|-----------|
| **Scope Alignment** | Is the vulnerable service in-scope? Was authorization confirmed? | Must be YES |
| **Impact Assessment** | Can this crash the service? Cause data loss? | Avoid destructive vectors in production |
| **Exploit Stability** | Is the exploit reliable? Or does it crash the target? | Prefer stable, well-tested exploits |
| **Detection Risk** | Will this trigger IDS/EDR/SIEM alerts? | Assess against engagement rules |
| **Authentication Required** | Does it need creds? Guest access? Anonymous? | Note access prerequisites |
| **Session Type** | Shell, service crash, info leak, or DoS? | Match to engagement objectives |
| **Privilege Level** | SYSTEM, Administrator, User, or Service context? | Higher privilege = more value |
| **Lateral Movement** | Can this enable pivoting to other hosts? | Document network implications |

### Decision Flow
```
IF (in-scope AND stable-exploit AND acceptable-risk)
    → Proceed to exploit planning
ELSE IF (unstable OR out-of-scope OR destructive)
    → Document and defer
ELSE
    → Flag for manual review
```
