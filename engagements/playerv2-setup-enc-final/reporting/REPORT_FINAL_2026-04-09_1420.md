## 3. Findings

### V-SETUP-001 — Encrypted installer remains independently unauditable before deployment

- **Severity:** High
- **CVSS Score:** N/A
- **Affected Target:** engagements/playerv2-artifacts/inbound/setup.enc
- **Description:** The assessed artifact is an OpenSSL salted password-encrypted container whose inner payload could not be validated across the full setup.enc engagement lineage. Reviewers could not confirm package contents, install logic, service behavior, persistence actions, or embedded secrets before any potential deployment decision.
- **Evidence:** ```
Confirmed file type: 'openssl enc'd data with salted password'; header marker 'Salted__' present; SHA-256 f9116a632d11c3b5331e5a9983caff074cc89ec7c8c63d5ac770794ec2d41b7b. Across rounds 1-8, no tested candidate family yielded a validated plaintext payload, recognizable file signature, or coherent installer content.
```
- **Impact:** The deployment trust decision remains blocked behind unavailable key material or unavailable workflow knowledge. Defenders and maintainers cannot independently inspect what the installer would do before use, increasing supply-chain, change-control, and incident-response risk.
- **Remediation:** Require a controlled decrypt-then-review workflow before any installation or execution. Recover the legitimate passphrase or original setup.enc generation workflow, decrypt only in an isolated analysis directory, identify the inner payload type, and complete a second-stage security review before deployment.
- **Hardening:** Treat encrypted setup blobs as opaque containers rather than trusted installers. Pair them with explicit release review gates, secure key custody, and evidence-backed validation of decrypted contents before operational use.
- **References:** engagements/playerv2-setup-enc/recon/RECON_SUMMARY_2026-04-08_1729.md, engagements/playerv2-setup-enc-round8/vuln/VULN_SUMMARY_2026-04-09_1408.md

### V-SETUP-002 — Companion trust metadata was not observed for the encrypted installer artifact

- **Severity:** Medium
- **CVSS Score:** N/A
- **Affected Target:** engagements/playerv2-artifacts/inbound/setup.enc
- **Description:** The encrypted setup artifact was assessed without a detached signature, signed manifest, or independent integrity metadata in the reviewed evidence set. Encryption was present, but an externally verifiable provenance chain was not observed alongside the blob.
- **Evidence:** ```
Enumeration in the initial and round2 engagements documented that no companion manifest, checksum package, or detached signature was present next to the artifact in the current evidence set.
```
- **Impact:** Operators cannot independently verify what the encrypted blob is supposed to contain or whether it matches an expected release before decryption. This increases provenance ambiguity and tamper-detection blind spots.
- **Remediation:** Distribute every encrypted installer with detached signatures, a signed manifest, or an independently verifiable checksum package. Preserve confidentiality and integrity as separate controls.
- **Hardening:** Adopt a release process that binds each encrypted artifact to versioned metadata, signer identity, expected hash values, and review status before installation decisions are made.
- **References:** engagements/playerv2-setup-enc/enum/ENUM_SUMMARY_2026-04-08_1729.md, engagements/playerv2-setup-enc-round2/enum/ENUM_SUMMARY_2026-04-08_1742.md

### V-SETUP-003 — Repeated false-positive decrypt conditions create analyst and operator misvalidation risk

- **Severity:** Medium
- **CVSS Score:** N/A
- **Affected Target:** setup.enc analysis workflow and any future recovery effort
- **Description:** Multiple rounds showed that OpenSSL wrong-key attempts can emit padding-valid or same-sized junk outputs that initially look promising even though no true plaintext was recovered. In round3, one sentence-style candidate returned a padding-valid result under aes-256-cbc legacy KDF, but secondary checks still showed high-entropy data rather than a decrypted installer.
- **Evidence:** ```
Round3 documented a padding-valid output for candidate 'AUTHORIZED_PI_SERIAL = "ffb6d42807368154"' that remained generic high-entropy data. Round4 and round7 further confirmed that incorrect-key attempts can still emit partial or replayed output files before 'bad decrypt', with no trustworthy archive, script, SQLite, ELF, JSON, XML, or HTML signature.
```
- **Impact:** Analysts may waste time, misreport success, or even treat garbage output as a legitimate installer if validation discipline is weak. This creates false confidence during incident response, maintenance, or security review.
- **Remediation:** Require strict validation gates before claiming decryption success: recognizable file signature or script header, coherent strings/metadata, stable file-type identification, and repeatable success with the same parameters. Treat OpenSSL exit-code success alone as non-authoritative for this artifact.
- **Hardening:** Document a formal decryption verification SOP for protected deployment artifacts so recovery and audit workflows cannot accept padding-only success, stdout junk, or replayed output files as proof of access.
- **References:** engagements/playerv2-setup-enc-round3/enum/ENUM_SUMMARY_2026-04-09_1015.md, engagements/playerv2-setup-enc-round4/enum/ENUM_SUMMARY_2026-04-09_1031.md, engagements/playerv2-setup-enc-round7/exploit/EXPLOIT_SUMMARY_2026-04-09_1216.md

### V-SETUP-004 — Recovery and audit depend on undisclosed decryption knowledge or unavailable workflow evidence

- **Severity:** Medium
- **CVSS Score:** N/A
- **Affected Target:** setup.enc operational key-handling and incident-response readiness
- **Description:** The setup.enc lineage repeatedly showed that neither the actual passphrase nor the original encryption/decryption workflow was available in the assessed evidence set. Evidence-backed serial-family, nearby phrase-family, expanded cipher/KDF matrix, and simple wrapper/preprocessing hypotheses all failed to produce validated plaintext.
- **Evidence:** ```
Failed targeted decrypt attempts were recorded across: serial-derived hypothesis family, setup-adjacent phrase family, cipher/KDF matrix coverage, replay-to-file validation, and wrapped candidate transforms (SHA-256/MD5/SHA-1/Base64 variants). Post-exploit review confirmed no payload access was obtained.
```
- **Impact:** Legitimate defenders may be unable to recover, inspect, or validate the installer during incident response or maintenance without privileged tribal knowledge. This creates a single-point-of-failure trust model and brittle operational recovery.
- **Remediation:** Recover and securely document the authoritative setup.enc generation/decryption procedure, including cipher/KDF profile, passphrase custody, approval path, and review workflow. If the current process cannot support accountable recovery, redesign it around managed secrets and documented release operations.
- **Hardening:** Move away from opaque, person-dependent decryption knowledge toward controlled key custody, documented recovery paths, and auditable release engineering practices.
- **References:** engagements/playerv2-setup-enc-round2/exploit/EXPLOIT_SUMMARY_2026-04-08_1742.md, engagements/playerv2-setup-enc-round6/vuln/VULN_SUMMARY_2026-04-09_1212.md, engagements/playerv2-setup-enc-round7/post-exploit/POST_EXPLOIT_SUMMARY_2026-04-09_1216.md, engagements/playerv2-setup-enc-round8/vuln/VULN_SUMMARY_2026-04-09_1408.md

## 4. Security Enhancement Recommendations

### Release Provenance and Integrity

Ship encrypted installer artifacts with detached signatures, signed manifests, and versioned checksum metadata so operators can verify provenance before decryption or deployment.

### Controlled Decrypt-and-Review Workflow

Require every protected installer to be decrypted only in an isolated analysis directory, then reviewed for payload type, scripts, services, persistence behavior, and secrets before execution on Raspberry Pi or production-adjacent systems.

### Key Custody and Recovery

Define who holds legitimate decryption material, how it is rotated, how authorized reviewers request access, and how recovery works during incident response without relying on chat-based password sharing or undocumented tribal knowledge.

### Decryption Validation SOP

Document mandatory plaintext validation gates so analysts cannot treat padding-valid OpenSSL output, partial stdout junk, or replayed output files as proof of successful decrypt.

### Security Architecture Review

If setup.enc is still required, review whether a safer packaging model can separate confidentiality, integrity, auditability, and deployment trust instead of bundling all trust into a single opaque encrypted blob.
