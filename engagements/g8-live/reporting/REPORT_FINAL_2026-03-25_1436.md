## 3. Findings

### VULN-002 — MySQL Database Service Exposed on Network

- **Severity:** Medium
- **CVSS Score:** 5.3
- **Affected Target:** 192.168.0.108:3306 (MySQL 8.4.8)
- **Description:** MySQL 8.4.8 is exposed on the network interface. While authentication is required, database services on workstations increase attack surface. Thread ID 50632 indicates active database usage, suggesting this is a production or development database.
- **Evidence:** ```
Nmap mysql-info script output:
- Version: 8.4.8
- Protocol: 10
- Thread ID: 50632 (suggests active connections)
- Auth Plugin: caching_sha2_password
- SSL: Supported (capabilities flag)
- Anonymous login: Denied (ERROR 1045)
- Root/no-password: Denied
```
- **Impact:** If credentials are compromised, an attacker could:
- Access sensitive data in MySQL databases
- Modify or delete database contents
- Potentially escalate privileges via MySQL UDF or stored procedures
- Use as pivot point for lateral movement within the network
- **Remediation:** Restrict MySQL network exposure:
1. Edit MySQL configuration: `sudo nano /etc/mysql/mysql.conf.d/mysqld.cnf` (Linux) or my.ini (Windows)
2. Set `bind-address = 127.0.0.1` to listen only on localhost
3. If remote access required, restrict via firewall: `New-NetFirewallRule -DisplayName 'MySQL Restrict' -Direction Inbound -LocalPort 3306 -RemoteAddress 192.168.0.0/24 -Action Allow`
4. Restart MySQL service
5. Verify: `netstat -an | findstr 3306` should show 127.0.0.1 only
- **Hardening:** Database hardening recommendations:
- Enforce SSL/TLS for all MySQL connections: `REQUIRE SSL` on user accounts
- Review and remove unnecessary MySQL user accounts
- Implement principle of least privilege for database users
- Enable MySQL audit logging: `SET GLOBAL general_log = 'ON'`
- Consider database firewall (MySQL Enterprise Firewall) or proxy
- Regular backup and test restoration procedures
- **References:** https://dev.mysql.com/doc/refman/8.4/en/security-against-attack.html, https://dev.mysql.com/doc/refman/8.4/en/encrypted-connections.html

### VULN-003 — SMB and RPC Services Exposed Without SMB Signing Enforcement

- **Severity:** Medium
- **CVSS Score:** 5.3
- **Affected Target:** 192.168.0.108:445 (SMB), 192.168.0.108:135 (RPC)
- **Description:** SMB service is accessible on the network. While NULL sessions and anonymous access are denied (good security posture), SMB signing enforcement could not be verified without authentication. Legacy SMB vulnerabilities (EternalBlue, Conficker, etc.) are confirmed patched.
- **Evidence:** ```
Enumeration results:
- NULL session: DENIED (NT_STATUS_ACCESS_DENIED)
- Anonymous enumeration: DENIED
- Guest access: DENIED
- SMB vulnerabilities tested: MS17-010, MS08-067, MS06-025, MS07-029, CVE-2009-3103, WebExec - ALL NOT VULNERABLE
- OS: Windows 11 24H2 (Build 26100)
```
- **Impact:** If SMB signing is not enforced:
- SMB relay attacks possible if NTLM authentication is captured
- Man-in-the-middle could modify SMB traffic
- Pass-the-hash attacks if credentials are compromised
Note: Target shows good security with anonymous access denied and all legacy CVEs patched.
- **Remediation:** Enforce SMB signing via Group Policy:
1. Open Group Policy Editor
2. Navigate to: Computer Configuration → Windows Settings → Security Settings → Local Policies → Security Options
3. Enable 'Microsoft network server: Digitally sign communications (always)'
4. Enable 'Microsoft network client: Digitally sign communications (always)'
5. Run: `gpupdate /force`
6. Verify with: `Get-SmbServerConfiguration | Select RequireSecuritySignature`
- **Hardening:** Additional SMB hardening:
- Disable SMBv1: `Set-SmbServerConfiguration -EnableSMB1Protocol $false`
- Ensure SMBv3 encryption is enabled: `Set-SmbServerConfiguration -EncryptData $true`
- Restrict SMB access to authorized subnets only
- Monitor SMB access logs for suspicious activity
- Consider disabling file sharing if not required
- **References:** https://docs.microsoft.com/en-us/windows-server/storage/file-server/smb-security, https://docs.microsoft.com/en-us/windows/security/threat-protection/security-settings/security-policy-settings

### VULN-001 — RDP NTLM Information Disclosure

- **Severity:** Low
- **CVSS Score:** 3.7
- **Affected Target:** 192.168.0.108:3389 (RDP Service)
- **Description:** The RDP service with NLA enabled leaks internal domain structure, hostname, and OS version via NTLM authentication handshake. This information disclosure occurs before successful authentication and reveals: NetBIOS domain name (NCTV360), DNS domain (nctv360.com), FQDN (G8.nctv360.com), and OS version (Windows 11 24H2 Build 26100).
- **Evidence:** ```
Nmap rdp-ntlm-info script output:
- Target_Name: NCTV360
- NetBIOS_Domain_Name: NCTV360
- DNS_Domain_Name: nctv360.com
- DNS_Computer_Name: G8.nctv360.com
- DNS_Tree_Name: nctv360.com
- Product_Version: 10.0.26100 (Windows 11 24H2)
```
- **Impact:** Information disclosure aids reconnaissance and potential credential-based attacks. Domain structure, hostname, and OS version can be used for targeted attacks, lateral movement planning, and timing attacks if Kerberos is in use.
- **Remediation:** Configure RDP to reduce information disclosure via Group Policy:
1. Open Group Policy Editor (gpedit.msc)
2. Navigate to: Computer Configuration → Administrative Templates → Windows Components → Remote Desktop Services → Remote Desktop Session Host → Security
3. Set 'Require use of specific security layer for remote (RDP) connections' to SSL/TLS
4. Enable 'Require user authentication for remote connections by using Network Level Authentication'
5. Apply policy and restart RDP service: `Restart-Service TermService -Force`
- **Hardening:** Additional hardening recommendations:
- Deploy RDP Gateway or VPN for remote access instead of exposing RDP directly
- Implement IP whitelisting for RDP at the firewall level
- Enable RDP session auditing and log monitoring
- Consider disabling RDP if not required: `Set-ItemProperty -Path 'HKLM:\System\CurrentControlSet\Control\Terminal Server' -name 'fDenyTSConnections' -value 1`
- **References:** https://docs.microsoft.com/en-us/windows/security/threat-protection/security-settings/network-level-authentication, https://docs.microsoft.com/en-us/windows-server/remote/remote-desktop-services/clients/rdp-files

## 4. Security Enhancement Recommendations

### Network Segmentation

Implement network segmentation to isolate development/production database services from the general user network. Consider placing database servers in a separate VLAN with restricted access controls. Apply firewall rules that block inbound connections to administrative services (MySQL 3306, SMB 445, RDP 3389) from user workstations.

### Patch Management

Maintain a regular patching schedule for both Windows OS and third-party software (MySQL). Subscribe to Oracle Critical Patch Updates for MySQL and Microsoft Security Bulletins. Test patches in a staging environment before production deployment. Consider automated patch management solutions.

### Access Control

Implement principle of least privilege across all services. For MySQL, review user accounts and remove unnecessary privileges. For SMB, restrict share access using NTFS and share permissions. For RDP, require VPN or bastion host access instead of direct exposure. Consider implementing MFA for administrative access.

### Monitoring and Logging

Enable and centralize security logs for all services. Configure Windows Event Forwarding for RDP and SMB access events. Enable MySQL general query log and audit plugin for connection tracking. Implement SIEM rules to alert on suspicious activity: multiple failed RDP logins, unusual MySQL connection patterns, SMB access from unexpected sources.

### D. Residual Risk Assessment

Following this assessment, the following residual risks remain:

1. **Credential-based attacks** — If weak credentials exist on MySQL, SMB, or RDP, an attacker with valid credentials could gain access to the system. Regular credential audits and strong password policies are recommended.

2. **Unverified patch status** — While major CVEs (EternalBlue, Conficker, etc.) are confirmed patched, the latest cumulative updates for Windows 11 Build 26100 were not verified. Ensure latest security updates are applied.

3. **Information disclosure** — Domain structure (nctv360.com) remains exposed via RDP NTLM handshake even after remediation of other issues. This is a protocol-level behavior requiring architectural changes (VPN/RDP Gateway) to fully mitigate.

4. **Network exposure** — Administrative services (MySQL, SMB, RDP) remain exposed on the network segment. If the network is compromised, these services provide attack surface.

**Overall Residual Risk:** MEDIUM — The target demonstrates good security practices (no anonymous access, NLA enabled, legacy CVEs patched), but credential compromise remains a concern. Network segmentation and credential security audits would reduce residual risk to LOW.
