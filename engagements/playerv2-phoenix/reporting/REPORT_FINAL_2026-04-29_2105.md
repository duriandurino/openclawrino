## 6. Detailed Findings

### PHX-V02 — Fail-open local access after hardware-check crash

- **Severity:** High
- **CVSS Score:** 7.3
- **CVSS Version:** 4.0
- **CVSS Label:** CVSS-B
- **CVSS Vector:** `CVSS:4.0/AV:P/AC:L/AT:N/PR:N/UI:N/VC:H/VI:H/VA:L/SC:N/SI:N/SA:N`
- **CVSS Rationale:** Physical access is required, but the attacker does not need existing credentials or user participation. Once the enforcement component crashes, the device preserves local shell and GUI access instead of failing safely, resulting in direct confidentiality and integrity exposure on the vulnerable system.
- **CVSS Assumptions:** Base-only scoring was used. Threat and Environmental metrics were not enriched. This finding is intentionally kept separate from PHX-V01 and scored as the fail-open consequence after the authorization path crashes.
- **Scoring Note:** Base-only scoring was used. Threat and Environmental metrics were not enriched. This finding is intentionally kept separate from PHX-V01 and scored as the fail-open consequence after the authorization path crashes.
- **Affected Target:** playerv2-phoenix device, hardware-check.service, local GUI and shell access paths
- **Description:** When Phoenix authorization logic encountered the missing mmcblk0 CID path, the device did not fail safely. Instead, the lock enforcement component crashed and preserved interactive local access to the underlying OS environment.
- **Evidence:** ```
Direct GUI and terminal observation on the target display, whoami returned pi, systemctl status hardware-check.service and journalctl output showed repeated crash on the missing CID path rather than an enforced lock state.
```
- **Impact:** A local attacker can preserve a usable foothold on the player after the authorization component crashes. In this engagement, that foothold enabled access to shell history, trust inputs, service logic, vault metadata, and later extension into protected-vault access and secret recovery.
- **Remediation:** Implement explicit fail-closed behavior for all authorization-check exceptions. Force the device into a minimal locked state when hardware identity cannot be validated. Block local shell and GUI access until an authenticated recovery procedure completes.
- **Hardening:** Isolate the lock path from nonessential UI startup behavior. Prevent interactive local sessions from appearing before authorization completes. Add integration tests that validate crash handling, missing-path handling, and dependency-failure handling.

### PHX-V03 — Sensitive provisioning artifact exposure in shell history

- **Severity:** High
- **CVSS Score:** 7.1
- **CVSS Version:** 4.0
- **CVSS Label:** CVSS-B
- **CVSS Vector:** `CVSS:4.0/AV:L/AC:L/AT:N/PR:L/UI:N/VC:H/VI:H/VA:L/SC:N/SI:N/SA:N`
- **CVSS Rationale:** The attacker needs local access to the shell history context, which in this engagement was realistically obtainable after the fail-open path. No extra victim interaction is needed, and the recovered provisioning command exposes operationally useful secret material and deployment workflow details that may support further compromise.
- **CVSS Assumptions:** Base-only scoring was used. Threat and Environmental metrics were not enriched. This score assumes the exposed provisioning passphrase and command structure remain operationally useful enough to support attack chaining, as judged from current engagement context.
- **Scoring Note:** Base-only scoring was used. Threat and Environmental metrics were not enriched. This score assumes the exposed provisioning passphrase and command structure remain operationally useful enough to support attack chaining, as judged from current engagement context.
- **Affected Target:** /home/pi/.bash_history, provisioning workflow, related setup secrets and deployment material
- **Description:** Local shell history retained a plaintext provisioning command containing a remote setup host and passphrase. Because PHX-V02 preserved local shell access, that history became directly accessible during the engagement.
- **Evidence:** ```
sed inspection of /home/pi/.bash_history revealed the plaintext provisioning command and passphrase. stat on the history file placed the artifact in the current operational period.
```
- **Impact:** A local attacker with shell access can recover secret-bearing provisioning material, deployment workflow knowledge, and a potentially reusable setup passphrase. This may enable reinstallation abuse, recovery-path abuse, environment cloning, or related attack chaining.
- **Remediation:** Remove secret-bearing provisioning commands from shell history. Rotate any setup passphrases, bootstrap commands, and related deployment credentials that may still be valid. Move provisioning secrets into protected, noninteractive channels.
- **Hardening:** Disable or strictly control shell history for privileged provisioning workflows. Use one-time, device-bound, or short-lived bootstrap material instead of static passphrases. Audit historical images and field workflows for similar retained provisioning artifacts.

### PHX-V01 — Storage-interface-dependent authorization failure

- **Severity:** Medium
- **CVSS Score:** 6.8
- **CVSS Version:** 4.0
- **CVSS Label:** CVSS-B
- **CVSS Vector:** `CVSS:4.0/AV:P/AC:L/AT:N/PR:N/UI:N/VC:H/VI:L/VA:L/SC:N/SI:N/SA:N`
- **CVSS Rationale:** The attacker needs physical access to present the same protected media through a USB SD path, but no prior privileges or victim interaction are required. The bypass undermines the intended authorization boundary and can expose sensitive local system access conditions on the vulnerable device itself.
- **CVSS Assumptions:** Base-only scoring was used. Threat and Environmental metrics were not enriched. This score treats the vulnerability as the authorization-bypass condition itself and keeps the more direct fail-open access consequence in PHX-V02.
- **Scoring Note:** Base-only scoring was used. Threat and Environmental metrics were not enriched. This score treats the vulnerability as the authorization-bypass condition itself and keeps the more direct fail-open access consequence in PHX-V02.
- **Affected Target:** playerv2-phoenix device, hardware-check.service, vault-mount.service
- **Description:** Phoenix authorization relied on a fixed Linux storage identity assumption rather than a transport-independent identity check for the protected media. The same trusted microSD card was accepted into a materially different runtime state when presented through an external USB SD reader instead of the Pi's native SD slot.
- **Evidence:** ```
Same Phoenix player microSD booted successfully through a USB SD reader, local commands showed the live root and boot media as /dev/sda2 and /dev/sda1, hardware_lock.py and unlock_vault.py both hardcoded /sys/block/mmcblk0/device/cid, and hardware-check.service crashed because the expected mmcblk0 path did not exist in USB-presented mode.
```
- **Impact:** A local attacker with physical device access can bypass the intended hardware-bound authorization condition by changing only how the same media is presented to the system. This weakens the local trust boundary and creates the conditions for a more serious fail-open foothold.
- **Remediation:** Refactor authorization logic to identify the protected media through stable device attributes independent of Linux block-device naming. Validate the actual trusted media identity across supported presentation modes. Treat inability to confirm the trusted media identity as a secure denial condition.
- **Hardening:** Centralize media-identity handling so trust logic is not duplicated across multiple scripts. Add regression tests for alternate storage presentation paths, including USB-attached SD readers. Add explicit negative-path tests that prove identity-read failures fail closed.

## 8. Security Enhancement Recommendations

### Architecture

Redesign trusted-media validation to be transport-independent and centralized rather than duplicated across multiple scripts.

### Failure handling

Implement deterministic fail-closed behavior whenever media identity cannot be resolved or validated.

### Secret management

Rotate and remove retained provisioning secrets, and move bootstrap workflows away from plaintext shell-history exposure.

### Vault and key protection

Reassess vault key derivation, DB key placement, and private-key storage so a local foothold cannot reproduce protected access from product-resident inputs alone.

### Recovery hardening

Require authenticated and signed recovery sources before any repair or restore workflow can modify protected media.
