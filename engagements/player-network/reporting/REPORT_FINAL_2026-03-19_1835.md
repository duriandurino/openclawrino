## 3. Findings

### VULN-001 — Unauthenticated Player API

- **Severity:** Critical
- **CVSS Score:** 9.8
- **Affected Target:** 192.168.0.161:3215 (Player fleet pattern)
- **Description:** A Node.js Express API exposed device information, playlists, content URLs, and update-related endpoints without any authentication. Confirmed endpoints included /ping, /api/device, /api/playlist, /api/content, and /api/update.
- **Evidence:** ```
curl -s http://192.168.0.161:3215/api/device
curl -s http://192.168.0.161:3215/api/playlist
curl -s http://192.168.0.161:3215/api/content
```
- **Impact:** Any attacker on the same network can enumerate players, collect operational data, extract content URLs, and map the fleet without credentials.
- **Remediation:** Restrict the API to localhost or a management interface only, enforce authentication and authorization on all endpoints, and block port 3215 from untrusted networks using host firewall rules or segmentation.
- **Hardening:** Implement mutual authentication between the Electron app and backend API, add API gateway controls, and continuously scan fleet builds to ensure local-only services are not exposed on the LAN.
- **References:** CWE-306, OWASP API1:2023

### VULN-002 — Public CloudFront CDN Access

- **Severity:** High
- **CVSS Score:** 7.5
- **Affected Target:** d193aoclwas08l.cloudfront.net
- **Description:** Ad and media content hosted behind CloudFront was publicly downloadable when the object URL was known, allowing direct unauthenticated retrieval of signage content.
- **Evidence:** ```
Direct HTTP GET requests to discovered CloudFront object URLs returned content without authentication.
```
- **Impact:** Attackers can steal media assets, analyze campaign material, and potentially support broader supply-chain or defacement planning.
- **Remediation:** Require signed URLs or signed cookies for CDN delivery, reduce object exposure windows, and ensure origin access is restricted to trusted delivery paths.
- **Hardening:** Separate public marketing assets from operational signage content, add access logging and alerting for bulk downloads, and rotate content URLs when exposure is suspected.
- **References:** AWS CloudFront signed URLs documentation

### VULN-003 — Backend Server Service Exposure

- **Severity:** High
- **CVSS Score:** 8.1
- **Affected Target:** 192.168.0.103 (MySQL, RDP, WinRM, AnyDesk, APIs)
- **Description:** The backend Windows server exposed multiple management and data services directly on the same flat network as player devices, including MySQL, RDP, WinRM, AnyDesk, and application APIs.
- **Evidence:** ```
Nmap service discovery identified 17+ open services on 192.168.0.103, including 3306/3307, 3389, 5985, 7070, 82, and 42052.
```
- **Impact:** A compromise of any internal foothold can be leveraged to target administrative services, attempt credential attacks, and expand to server takeover or data exfiltration.
- **Remediation:** Move backend management services to a restricted admin network or VPN-only segment, close unnecessary ports, and allow access only from explicitly approved management hosts.
- **Hardening:** Deploy host-based firewall policy, enforce tiered network segmentation between players and servers, and require MFA or bastion access for remote administration.
- **References:** Microsoft WinRM hardening guidance, CIS Network Segmentation guidance

### VULN-004 — OAuth2 Authentication Flow Weakness

- **Severity:** High
- **CVSS Score:** 7.2
- **Affected Target:** 192.168.0.103:42052/api/auth/login
- **Description:** An OAuth2 authorization-code style flow was active on the backend API. While not directly exploitable without valid codes, its exposure on the internal network increases risk if codes or redirect paths are intercepted or abused.
- **Evidence:** ```
Backend auth endpoint at 192.168.0.103:42052/api/auth/login confirmed active during testing.
```
- **Impact:** If valid authorization codes or redirect handling weaknesses are abused, attackers may achieve account takeover or obtain backend API access.
- **Remediation:** Restrict auth endpoints to intended trust boundaries, validate redirect URIs strictly, shorten code lifetimes, and bind authorization flows to PKCE and trusted clients only.
- **Hardening:** Add anomaly detection for OAuth exchanges, require secure client registration controls, and isolate identity endpoints from player-accessible network segments.
- **References:** OAuth 2.0 Security Best Current Practice, RFC 7636 (PKCE)

### VULN-005 — CORS Misconfiguration

- **Severity:** Medium
- **CVSS Score:** 5.3
- **Affected Target:** 192.168.0.103:82
- **Description:** The backend Kestrel service presented permissive cross-origin behavior, including Access-Control-Allow-Origin: * in a context that could support unsafe browser-based interactions.
- **Evidence:** ```
HTTP responses from the service exposed permissive CORS headers during manual testing.
```
- **Impact:** Malicious websites visited by a user on the same network may be able to trigger cross-origin requests against the backend API depending on browser/client behavior and authentication state.
- **Remediation:** Replace wildcard origins with an explicit allowlist, disable credentialed cross-origin access where not required, and review all CORS middleware defaults.
- **Hardening:** Add automated security header testing to CI/CD, standardize a secure reverse-proxy policy for all APIs, and monitor for unexpected Origin headers in logs.
- **References:** OWASP CORS Misconfiguration guidance

### VULN-006 — VNC Service Exposed on Player

- **Severity:** Medium
- **CVSS Score:** 5.9
- **Affected Target:** 192.168.0.161:5900
- **Description:** A VNC service was reachable on the player device. Testing showed authentication rate limiting, but the service remained exposed and reachable to the local network.
- **Evidence:** ```
Port 5900 was open and reachable; authentication attempts triggered rate limiting after repeated tries.
```
- **Impact:** If credentials are discovered elsewhere or controls weaken, attackers may gain direct visual/interactive access to the player.
- **Remediation:** Disable VNC where not operationally required, bind remote desktop services to a management-only network, and enforce strong authentication plus IP restrictions.
- **Hardening:** Require remote support through a brokered support channel instead of direct LAN exposure, and alert on repeated failed VNC authentication attempts.
- **References:** CIS Raspberry Pi hardening guidance

## 4. Security Enhancement Recommendations

### Network Segmentation

Separate player devices, backend servers, and administrative workstations into distinct VLANs or firewall zones. Player subnets should not have direct lateral access to backend administration ports.

### Service Exposure Reduction

Audit every listening service on players and backend hosts and remove or firewall services that are not essential for the runtime mission of each device.

### Application Security

Treat internal APIs as hostile-by-default: require authentication, authorization, input validation, and explicit trust boundaries even for services primarily consumed by local desktop applications.

### Monitoring and Detection

Log and alert on port scans, repeated authentication attempts, bulk CDN downloads, unusual API enumeration, and access to management endpoints from non-admin segments.
