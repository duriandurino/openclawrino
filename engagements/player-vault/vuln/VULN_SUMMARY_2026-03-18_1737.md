## Vulnerability Inventory

### V-012: Hardcoded Encryption Key in Hardware Serial
**Status:** CONFIRMED (Exploited)
**Severity:** CRITICAL (CVSS 9.1)

#### Details
| Property | Value |
|----------|-------|
| **File** | `/usr/local/bin/hardware_lock.py` |
| **Vulnerable Code** | `AUTHORIZED_PI_SERIAL = "ffb6d42807368154"` |
| **Exploit File** | `/usr/local/bin/unlock_vault.py` |
| **Target** | `/home/pi/vault.img` (LUKS2 encrypted) |
| **Discovery Method** | Source code analysis |
| **Exploitation** | `echo "ffb6d42807368154" | sudo cryptsetup open vault.img nctv_data -` |

#### CVSS 3.1 Vector
`AV:L/AC:L/PR:L/UI:N/S:U/C:H/I:H/A:H`

#### Description
The vault encryption uses the Raspberry Pi's hardware serial number as the LUKS passphrase. This serial is:
- Hardcoded in `hardware_lock.py`
- Used in `unlock_vault.py` without additional key derivation
- Discoverable via `cat /proc/cpuinfo`
- Logged in `.bash_history`

#### Impact
- Full vault decryption
- Access to player-server source code
- Access to database backups (db_backup_dirty/)
- Access to application logs (potential credential leakage)
- Supply chain attack vector (modify server code)

#### Evidence
```bash
# Discovery from .bash_history
/home/pi/.bash_history:1191:echo "ffb6d42807368154" | sudo cryptsetup luksFormat /home/pi/vault.img -
/home/pi/.bash_history:1192:echo "ffb6d42807368154" | sudo cryptsetup open /home/pi/vault.img vault_data -
/home/pi/.bash_history:1194:echo -n "ffb6d42807368154" | sudo cryptsetup luksFormat /home/pi/vault.img --key-file -
/home/pi/.bash_history:1195:echo -n "ffb6d42807368154" | sudo cryptsetup open /home/pi/vault.img vault_data --key-file -

# Successful exploitation
echo "ffb6d42807368154" | sudo cryptsetup open /home/pi/vault.img nctv_data -
# Result: SUCCESS
```

### V-013: Newline Character Key Derivation Weakness
**Status:** CONFIRMED (Exploited)
**Severity:** HIGH (CVSS 7.5)

#### Details
| Property | Value |
|----------|-------|
| **Issue** | Key derivation includes trailing newline from `echo` |
| **Impact** | Key differs based on command syntax (`echo` vs `echo -n`) |
| **Discovery** | Failed attempts using `--key-file -` with `echo -n` |

#### Description
The vault was originally formatted using `echo "serial" | cryptsetup ...` which includes a newline character (`\n`) at the end. Using `echo -n "serial" | cryptsetup ... --key-file -` (without newline) produces a different key and fails. This inconsistency suggests poor key management practices.

#### CVSS 3.1 Vector
`AV:L/AC:L/PR:L/UI:N/S:U/C:H/I:H/A:N`

### V-014: Security Through Obscurity (Device Binding)
**Status:** CONFIRMED
**Severity:** MEDIUM (CVSS 5.3)

#### Details
| Property | Value |
|----------|-------|
| **Mechanism** | Hardware serial binding for vault unlock |
| **Assumption** | Physical access required to discover serial |
| **Bypass** | Serial discoverable from source code |

#### Description
The developers implemented device-binding by using the Pi's hardware serial as the vault key. This was intended to prevent vault access on unauthorized devices. However, the serial is discoverable through:
1. Source code (`hardware_lock.py`)
2. Command history (`.bash_history`)
3. System files (`/proc/cpuinfo`)
4. The unlock script itself (`unlock_vault.py`)

This is "security through obscurity" — the protection relies on the serial being unknown, not on cryptographic strength.
