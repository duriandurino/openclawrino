## 1. Executive Summary

During this penetration test, we successfully decrypted a LUKS2-encrypted vault container (`vault.img`) on a Raspberry Pi 5 digital signage player in approximately **15 minutes** using only information discovered through local filesystem analysis.

### The Core Vulnerability

The vault's encryption was **cryptographically strong** — AES-XTS-plain64 with a 512-bit key and argon2id key derivation function. However, the key management implementation was fundamentally flawed:

1. **The encryption passphrase was hardcoded** in a Python script (`hardware_lock.py`)
2. **The passphrase was derived from easily discoverable hardware** — the Raspberry Pi's serial number
3. **The passphrase was exposed in shell history** (`.bash_history`)

### What Was Exposed

Upon decryption, we gained access to:

| Category | Details |
|----------|---------|
| **Application Source Code** | Full Node.js backend (player-server v2.9.43-pi5) |
| **Database** | 86KB SQLite database with 18 tables of operational data |
| **API Credentials** | Vistar ad network API keys in plaintext |
| **License Keys** | Device and content licensing information |
| **Core Dumps** | 8 crash dump files (~2.6 GB total) containing process memory |
| **Configuration** | Environment variables, PM2 process definitions |

### Key Metrics

| Metric | Value |
|--------|-------|
| Time to decrypt | ~15 minutes |
| Access required | Local filesystem read |
| Credentials exposed | API keys, license keys |
| Data exposure | ~2.6 GB + operational database |
| **Overall Risk Rating** | **CRITICAL** |

## 4. Technical Background

### 4.1 LUKS2 Encryption Overview

LUKS (Linux Unified Key Setup) is the standard disk encryption specification for Linux. LUKS2 is the second version, offering:

- **Multiple key slots** (8 in this case) — allows multiple passphrases
- **Advanced key derivation** — argon2id (resistant to GPU/ASIC attacks)
- **Sector size flexibility** — 4096 bytes in this implementation
- **Strong cipher defaults** — AES-XTS-plain64 with 512-bit keys

### 4.2 Why This Attack Worked

Despite strong cryptography, the vault was vulnerable because:

1. **Key Management Failure:** The passphrase was derived from hardware serial, not a secret known only to authorized users
2. **Hardcoded Credentials:** The authorized serial was stored in plaintext source code
3. **Poor Operational Security:** Shell history retained the exact format command used for encryption
4. **Security Through Obscurity:** The developers assumed physical access would prevent serial number discovery

### 4.3 The Newline Character Factor

A critical technical detail emerged during decryption:

| Method | Command | Result |
|--------|---------|--------|
| **FAILED** | `echo -n "ffb6d42807368154" \| cryptsetup ... --key-file -` | No key available |
| **SUCCEEDED** | `echo "ffb6d42807368154" \| cryptsetup open vault.img nctv_data -` | SUCCESS |

The original encryption command (found in `.bash_history`) used `echo` (with trailing newline), not `echo -n`. This meant the passphrase technically included a `\n` character at the end.

## 5. Attack Chain

### 5.1 Stage 1: Discovery

**Objective:** Locate and identify the encrypted vault

**Actions Taken:**
```bash
# Located vault in home directory
ls -la /home/pi/vault.img

# Identified LUKS2 encryption
file /home/pi/vault.img
# Output: LUKS encrypted file, ver 2, UUID: 9757eca5-e8a1-4f8a-9c20-8c9252d61d09

# Analyzed encryption parameters
sudo cryptsetup luksDump vault.img
```

**Findings:**
- LUKS2 container with AES-XTS-plain64, 512-bit key
- argon2id PBKDF (time cost: 4, memory: 850368, threads: 4)
- Single key slot in use (slot 0 of 8)
- Not currently mounted or unlocked

### 5.2 Stage 2: Key Discovery

**Objective:** Determine the encryption passphrase

**Actions Taken:**
1. Discovered `unlock_vault.py` in `/usr/local/bin/`
2. Analyzed script — revealed Pi serial number as passphrase source
3. Found `hardware_lock.py` with hardcoded authorized values
4. Confirmed passphrase format via `.bash_history` analysis

**Key Discovery:**
```python
# From hardware_lock.py
AUTHORIZED_PI_SERIAL = "ffb6d42807368154"
AUTHORIZED_SD_CID = "1b534d45423151543089c65df4015a00"

# From .bash_history (line 1191-1192)
echo "ffb6d42807368154" | sudo cryptsetup luksFormat /home/pi/vault.img -
echo "ffb6d42807368154" | sudo cryptsetup open /home/pi/vault.img vault_data -
```

**Critical Insight:** The passphrase was NOT this device's serial (`dadf6f9ef35e55ab`) but the AUTHORIZED serial from the source code (`ffb6d42807368154`).

### 5.3 Stage 3: Decryption

**Objective:** Unlock the LUKS container and mount the filesystem

**Actions Taken:**
```bash
# Close any existing device
sudo cryptsetup close nctv_data 2>/dev/null

# Open vault using discovered passphrase (WITH newline via echo)
echo "ffb6d42807368154" | sudo cryptsetup open /home/pi/vault.img nctv_data -

# Mount decrypted filesystem
sudo mount /dev/mapper/nctv_data /home/pi/n-compasstv-secure
```

**Result:** SUCCESS — vault mounted at `/home/pi/n-compasstv-secure/`

### 5.4 Stage 4: Data Exfiltration

**Objective:** Inventory and analyze vault contents

**Actions Taken:**
- Listed all directories and files
- Examined database schema (18 tables)
- Identified sensitive files (.env, config.js, core dumps)
- Documented exposed credentials

## 6. Detailed Findings

### Finding V-012: Hardcoded Encryption Key

| Property | Value |
|----------|-------|
| **Severity** | CRITICAL |
| **CVSS v3.1** | 9.1 |
| **Vector** | AV:L/AC:L/PR:L/UI:N/S:U/C:H/I:H/A:H |
| **Category** | CWE-798: Use of Hard-coded Credentials |

**Description:**
The LUKS vault encryption passphrase is the Raspberry Pi's hardware serial number, which is:
- Hardcoded in `hardware_lock.py` as `AUTHORIZED_PI_SERIAL = "ffb6d42807368154"`
- Used directly as the LUKS passphrase without additional key derivation
- Discoverable through multiple paths (source code, /proc/cpuinfo, .bash_history)

**Evidence:**
```python
# hardware_lock.py line 6
AUTHORIZED_PI_SERIAL = "ffb6d42807368154"

# unlock_vault.py lines 13-15
serial = get_pi_serial()
cmd = f"echo -n '{serial}' | sudo cryptsetup open {vault} nctv_data --key-file -"
subprocess.run(cmd, shell=True)
```

**Impact:**
- Full vault decryption by any user with filesystem read access
- Access to all application source code, database, and credentials
- Supply chain attack potential (modify player-server code)

**Remediation:**
1. **Immediate:** Change the vault encryption passphrase to a randomly generated value (minimum 32 characters)
2. **Short-term:** Remove all hardcoded credentials from source code
3. **Long-term:** Implement TPM-based key storage or hardware security module

### Finding V-013: Newline Character Key Derivation Weakness

| Property | Value |
|----------|-------|
| **Severity** | HIGH |
| **CVSS v3.1** | 7.5 |
| **Vector** | AV:L/AC:L/PR:L/UI:N/S:U/C:H/I:H/A:N |
| **Category** | CWE-330: Use of Insufficiently Random Values |

**Description:**
The vault passphrase includes a trailing newline character (`\n`) due to the use of `echo` instead of `echo -n` during encryption setup. This inconsistency:
- Causes confusion between `--key-file -` and standard input methods
- Indicates poor key management practices
- May lead to failed decryption attempts if the newline is missed

**Evidence:**
```bash
# Encryption command (from .bash_history)
echo "ffb6d42807368154" | sudo cryptsetup luksFormat /home/pi/vault.img -

# Decryption with --key-file (FAILED)
echo -n "ffb6d42807368154" | sudo cryptsetup open vault.img nctv_data --key-file -
# Result: No key available with this passphrase

# Decryption without --key-file (SUCCEEDED)
echo "ffb6d42807368154" | sudo cryptsetup open /home/pi/vault.img nctv_data -
# Result: SUCCESS
```

**Remediation:**
1. Re-encrypt the vault using a proper keyfile with consistent binary format
2. Document the exact passphrase format for future reference
3. Use `cryptsetup luksAddKey` with a keyfile instead of inline echo

### Finding V-014: Security Through Obscurity (Device Binding)

| Property | Value |
|----------|-------|
| **Severity** | MEDIUM |
| **CVSS v3.1** | 5.3 |
| **Vector** | AV:L/AC:L/PR:L/UI:N/S:U/C:L/I:L/A:L |
| **Category** | CWE-656: Reliance on Security Through Obscurity |

**Description:**
The developers implemented device-binding by using the Pi's hardware serial as the vault key. This was intended to prevent vault access on unauthorized devices. However, the "security" relies entirely on the serial being unknown — not on cryptographic strength.

**Analysis:**
| Assumption | Reality |
|------------|---------|
| Serial is hardware-specific and secret | Serial is easily discoverable from /proc/cpuinfo |
| Only physical access reveals serial | Serial is also hardcoded in source code |
| Device binding prevents unauthorized access | Any user can read and reuse the serial |

**Remediation:**
1. Do not rely on hardware identifiers as cryptographic keys
2. Implement proper passphrase-based encryption with user-generated secrets
3. If device binding is required, use TPM or secure boot mechanisms

## 10. Conclusion

This engagement demonstrated that **strong cryptography is only as secure as its key management**. Despite using AES-XTS-512 with argon2id PBKDF — both cryptographically sound choices — the vault was compromised in approximately 15 minutes using a plaintext serial number found in a Python script.

### Key Takeaways

1. **Never hardcode credentials** in source code, especially encryption keys
2. **Hardware serials are not secrets** — they are easily discoverable and should never be used as cryptographic keys
3. **Shell history retention** can expose sensitive commands and credentials
4. **Development mode in production** weakens security posture
5. **Unencrypted databases** are a critical liability in compromised environments

### Risk Rating Justification

The **CRITICAL** overall risk rating is justified because:
- A single local filesystem compromise leads to complete vault decryption
- Exposed credentials could enable supply chain attacks across the ad network
- License theft could result in significant financial impact
- The attack requires minimal skill and no special tools

### Next Steps

1. Implement immediate remediation actions (P0 items)
2. Schedule follow-up assessment after remediation
3. Review all Player devices fleet-wide for similar vulnerabilities
4. Consider mandatory security training for development team
