## 1. Executive Summary

During this engagement, the security team successfully decrypted a LUKS2-encrypted vault container (`vault.img`) on a Raspberry Pi 5 digital signage player without the owner's knowledge of the extraction path. The entire attack chain — from discovery to full data exfiltration — required **only local filesystem access and publicly available system information** (the Pi's hardware serial number).

**The vault's encryption was cryptographically strong (AES-XTS-plain64, 512-bit, argon2id KDF), but its key management was fundamentally broken.** The passphrase was:

1. **Hardcoded** in a Python script (`hardware_lock.py`)
2. **Derived from easily discoverable hardware** (Pi serial number)
3. **Exposed in shell history** (`.bash_history`)

Once the vault was decrypted, we obtained access to:

- **An entire Node.js application** (player-server v2.9.43-pi5) including full source code
- **An unprotected SQLite database** containing 18 tables of operational data
- **Third-party API keys** (Vistar ad network credentials)
- **License keys** in plaintext
- **8 core dump files** (~2.6 GB total) containing potential process memory snapshots
- **Application configuration** including environment variables and PM2 process definitions

### Key Takeaways

| Metric | Value |
|--------|-------|
| Time to decrypt vault | ~15 minutes (manual) |
| Required access level | Any user with filesystem read access |
| Credentials exposed | API keys, license keys, Pi serial |
| Data exposure | ~2.6 GB core dumps + operational database |
| Critical findings | 1 (CVSS 9.1) |
| High findings | 4 (CVSS 7.5 – 8.1) |
| Medium findings | 4 (CVSS 5.3 – 6.5) |
| **Overall risk** | **CRITICAL** |

## 3. Target Overview

### 3.1 Vault Configuration

The `vault.img` file is a LUKS2-format encrypted disk image stored at `/home/pi/vault.img`. Upon analysis via `cryptsetup luksDump`, the following cryptographic parameters were confirmed:

| Parameter | Value |
|-----------|-------|
| Version | LUKS2 |
| Cipher | aes-xts-plain64 |
| Key Size | 512 bits (256-bit effective for XTS) |
| KDF | argon2id |
| UUID | 9757eca5-e8a1-4f8a-9c20-8c9252d61d09 |
| Segments | 1 (LUKS2 key slot 0 active) |

**Cryptographic Assessment:** The encryption algorithm itself is strong and industry-standard. AES-XTS with 512-bit key and argon2id KDF represents best practices for disk encryption. **The vulnerability is entirely in key management, not cryptography.**

### 3.2 Supporting Scripts

Two Python scripts in `/usr/local/bin/` manage vault access:

- **`unlock_vault.py`** — Automated vault unlock script that reads the Pi serial and calls cryptsetup
- **`hardware_lock.py`** — Device authentication module containing the hardcoded serial

## 4. Attack Chain Narrative

The full attack chain from initial access to complete data compromise followed this path:

### Stage 1: Reconnaissance (Discovery)

```
Attacker gains read access to /home/pi/ filesystem
    ↓
Finds vault.img (LUKS2 encrypted, UUID identified)
    ↓
Discovers supporting scripts: unlock_vault.py, hardware_lock.py
    ↓
Identifies encryption type via cryptsetup luksDump
```

**No special tools required.** Standard Linux utilities (`ls`, `file`, `cryptsetup`) are sufficient.

### Stage 2: Credential Harvesting (Key Discovery)

```
Reviews unlock_vault.py → reveals vault uses Pi hardware serial as passphrase
    ↓
Reviews hardware_lock.py → finds AUTHORIZED_PI_SERIAL = "ffb6d42807368154"
    ↓
Checks .bash_history → confirms actual cryptsetup command used for encryption
    ↓
Key extracted: ffb6d42807368154 (with trailing newline)
```

**The Pi serial number is freely available** via `cat /proc/cpuinfo`, `vcgcenc --get_serial`, or physically printed on the board. The "hardware lock" provides zero additional security.

### Stage 3: Decryption (Exploitation)

```
Attempt 1: echo -n "ffb6d42807368154" | cryptsetup open vault.img nctv_data --key-file -
    → FAILED (wrong key — missing newline)
    
Attempt 2: echo "ffb6d42807368154" | cryptsetup open /home/pi/vault.img nctv_data
    → SUCCESS (correct key — echo adds trailing \n)
    
Mount: mount /dev/mapper/nctv_data /home/pi/n-compasstv-secure
    → Full filesystem access granted
```

**Critical detail:** The newline character (`\n`) appended by `echo` (vs `echo -n`) is part of the passphrase. This is both a quirk that adds minimal complexity and a weakness — it suggests the passphrase was set accidentally with the newline, meaning the *intended* key material is even shorter than expected.

### Stage 4: Data Exfiltration

```
Full filesystem tree enumerated
    ↓
SQLite database (_data.db) extracted and schema analyzed
    ↓
18 tables identified containing: API keys, license keys, operational data
    ↓
Core dump files (~2.6 GB) flagged for potential memory forensics
    ↓
Application source code and configuration extracted
```

### Attack Complexity Assessment

| Factor | Rating | Notes |
|--------|--------|-------|
| Skill required | Low | Copy-paste from discovered scripts |
| Time required | ~15 min | Manual; seconds with automation |
| Tools needed | Standard Linux | No exploits or special software |
| Trace of attack | Low | Read-only access leaves minimal footprint |
| Privilege needed | Any local user | No root required for read access |

## 5. Detailed Findings

### V-012: Hardcoded Encryption Key

| Attribute | Value |
|-----------|-------|
| **Severity** | **CRITICAL** |
| **CVSS v3.1** | **9.1** (AV:L/AC:L/PR:N/UI:N/S:C/C:H/I:H/A:N) |
| **Category** | Cryptographic Failure / Hardcoded Credentials |
| **Location** | `/usr/local/bin/hardware_lock.py` |

#### Description

The LUKS2 vault passphrase is hardcoded in plaintext in `hardware_lock.py` as the constant `AUTHORIZED_PI_SERIAL = "ffb6d42807368154"`. This value is directly used as the encryption key material for the vault container. Anyone with filesystem read access to this file can decrypt the entire vault.

#### Evidence

```python
# /usr/local/bin/hardware_lock.py (extracted)
AUTHORIZED_PI_SERIAL = "ffb6d42807368154"
```

The `.bash_history` further confirms this key was used during vault creation:

```bash
echo "ffb6d42807368154" | sudo cryptsetup luksFormat /home/pi/vault.img -
```

#### Impact

- **Complete vault compromise** — any attacker with read access can decrypt
- **No key rotation possible** without re-encrypting the entire container
- **Pi serial is public information** — printed on board, available via software
- Compromises **all data** within the vault (application code, database, credentials)

#### Remediation

**Immediate (within 24 hours):**
1. Re-encrypt `vault.img` with a randomly generated passphrase (minimum 32 characters)
2. Remove `hardware_lock.py` and `unlock_vault.py` or restructure to use a proper secrets manager
3. Clear `.bash_history` of the passphrase: `history -c && cat /dev/null > ~/.bash_history`
4. Rotate ALL credentials that were stored in the vault (API keys, license keys)

**Long-term:**
- Implement hardware-backed key storage (TPM 2.0, or at minimum, a separate encrypted partition for key material)
- Use `systemd-creds` or a secrets manager (HashiCorp Vault, AWS Secrets Manager) for key injection at boot
- Never derive cryptographic keys from hardware serial numbers

### V-013: Newline Key Derivation Weakness

| Attribute | Value |
|-----------|-------|
| **Severity** | **HIGH** |
| **CVSS v3.1** | **7.5** (AV:L/AC:L/PR:N/UI:N/S:U/C:H/I:N/A:N) |
| **Category** | Cryptographic Misconfiguration |
| **Location** | Vault passphrase mechanism |

#### Description

The vault passphrase includes a newline character (`\n`) as part of the key material. This occurred because `echo` (without `-n`) was used during vault creation, appending a trailing newline to the passphrase. This creates several issues:

1. **Unintentional key material** — the `\n` was almost certainly not a deliberate security decision
2. **Key fragility** — using `echo -n` (the more common/obvious form) produces a different key, making the vault inaccessible
3. **Reduced entropy perception** — the actual effective key material is the 16-character serial + 1 newline = 17 bytes, of which only 16 are non-trivial

#### Evidence

```
FAILED:  echo -n "ffb6d42807368154" | cryptsetup open vault.img nctv_data --key-file -
SUCCESS: echo "ffb6d42807368154" | cryptsetup open /home/pi/vault.img nctv_data
```

The newline character is consumed as part of the passphrase by `cryptsetup` when reading from stdin.

#### Impact

- Effective key space is only the 16-character Pi serial + 1 known character (`\n`)
- Brute force against this key space is trivial (16 hex digits = 2^64 possibilities, but Pi serials follow a known format)
- Anyone who discovers the serial but forgets the newline quirk cannot unlock the vault — but this is security by accident, not design

#### Remediation

1. Re-encrypt vault with an intentionally strong, random passphrase
2. Document and validate passphrase handling in any automation scripts (always specify whether trailing newlines are included)
3. Use `--key-file-size` or `--key-file-offset` parameters explicitly when feeding key material to cryptsetup

### V-014: Security Through Obscurity (Device Binding)

| Attribute | Value |
|-----------|-------|
| **Severity** | **MEDIUM** |
| **CVSS v3.1** | **5.3** (AV:L/AC:L/PR:N/UI:N/S:U/C:L/I:N/A:N) |
| **Category** | Weak Authentication / Security Design |
| **Location** | Architecture level — vault unlock mechanism |

#### Description

The vault's "hardware lock" mechanism binds access to the Pi's hardware serial number, creating a false sense of security through obscurity. The design assumes that because the serial is "hardware-specific," it constitutes a valid authentication factor. In reality:

- Pi serial numbers are **readable by any local user** (`/proc/cpuinfo`, `vcgcenc --get_serial`)
- Pi serial numbers are **printed on the physical board** (visible without disassembly)
- Pi serial numbers follow a **predictable format** (16 hex digits)
- There is **no attestation** — the serial is just a string, not a cryptographic proof of hardware identity

#### Evidence

The entire "security" of the vault rests on:

```python
AUTHORIZED_PI_SERIAL = "ffb6d42807368154"
```

This provides no more security than a password stored in a config file.

#### Impact

- Anyone with physical or remote access to the Pi can trivially obtain the serial
- An attacker who replaces the SD card with their own can still read the serial from the board
- False confidence in security may delay detection of compromise

#### Remediation

1. Do not rely on hardware serial numbers as cryptographic secrets
2. If hardware binding is required, implement proper hardware attestation (TPM 2.0 sealed keys)
3. Use multi-factor vault unlocking: passphrase + hardware token + integrity verification

### V-015: Development Mode Exposed in Production

| Attribute | Value |
|-----------|-------|
| **Severity** | **MEDIUM** |
| **CVSS v3.1** | **5.8** (AV:L/AC:L/PR:N/UI:R/S:U/C:L/I:L/A:H) |
| **Category** | Security Misconfiguration |
| **Location** | `/home/pi/n-compasstv-secure/player-server/.env` |

#### Description

The application's `.env` file sets `NODE_ENV=development` on a production digital signage player. Development mode typically enables:

- Verbose error messages and stack traces exposed to clients
- Disabled security middleware (CSRF, rate limiting, input validation)
- Hot module reloading and debug endpoints
- Relaxed CORS policies
- Detailed logging that may include sensitive data

#### Evidence

```
PORT=3215
NODE_ENV=development
```

Found in `player-server/.env` within the vault.

#### Impact

- Potential exposure of internal application state through error messages
- Debug endpoints may be accessible to remote users
- Reduced input validation may enable injection attacks
- Verbose logging may leak sensitive information to log files

#### Remediation

1. Change `.env` to `NODE_ENV=production` immediately
2. Audit application code for development-only features (debug routes, test endpoints, mock data)
3. Implement environment-based feature flags that disable debug functionality in production
4. Add a startup check that warns/fails if `NODE_ENV=development` in production builds

### V-016: Vistar API Keys in Database

| Attribute | Value |
|-----------|-------|
| **Severity** | **HIGH** |
| **CVSS v3.1** | **8.1** (AV:L/AC:L/PR:L/UI:N/S:C/C:H/I:H/A:N) |
| **Category** | Sensitive Data Exposure / Hardcoded Credentials |
| **Location** | `global_settings` table in `_data.db` |

#### Description

The `global_settings` table in the SQLite database contains API keys for the Vistar advertising network, including:

- `vistar_api_key` — Primary API authentication key
- `vistar_net_key` — Network-level API key
- `fast_edge_api` — Edge computing API key

These credentials are stored in plaintext within an unencrypted SQLite database, accessible to any process or user that can read the database file.

#### Evidence

```sql
-- global_settings table schema
Column: vistar_api_key   (API key for Vistar ad network)
Column: vistar_net_key   (Network-level key)
Column: fast_edge_api    (Edge API key)
```

#### Impact

- **Full account takeover** of the Vistar ad network account
- Ability to **manipulate ad content** displayed on signage network
- Potential **financial fraud** through fraudulent ad impressions
- Lateral access to other Vistar-connected systems
- Reputational damage if ads are replaced with malicious content

#### Remediation

1. Rotate all Vistar API keys immediately via the Vistar dashboard
2. Implement key vault integration — store API keys in an encrypted secrets manager, not the database
3. Use short-lived tokens with automatic refresh where possible
4. Add API key access auditing/logging to detect unauthorized use
5. Restrict API key permissions to minimum required scope (principle of least privilege)

### V-017: License Keys Stored in Plaintext

| Attribute | Value |
|-----------|-------|
| **Severity** | **HIGH** |
| **CVSS v3.1** | **7.5** (AV:L/AC:L/PR:L/UI:N/S:U/C:H/I:H/A:N) |
| **Category** | Sensitive Data Exposure |
| **Location** | `license` table in `_data.db` |

#### Description

The `license` table stores license keys in plaintext. Additional license-related configuration (reboot_time, CEC settings, screenshot settings, speedtest settings) is also stored without encryption. This allows anyone with database access to:

- Extract valid license keys for unauthorized use
- Modify license configuration (reboot schedules, feature flags)
- Potentially clone or revoke licenses for other devices

#### Evidence

```sql
-- license table schema
license_id, license_key, type_id, reboot_time,
cec_config, screenshot_config, speedtest_config
```

License keys are stored as plain strings with no hashing or encryption.

#### Impact

- License theft and unauthorized redistribution
- Potential revenue loss from license key sharing
- Ability to modify device behavior through license configuration tampering
- Denial of service to legitimate devices if licenses are revoked/modified

#### Remediation

1. Never store license keys in plaintext — hash them (SHA-256 minimum) for verification
2. Encrypt the entire SQLite database using SQLCipher or move to an encrypted database backend
3. Implement server-side license validation with signed tokens instead of local key storage
4. Add license binding to hardware attestation (TPM-based, not serial-based)

### V-018: Core Dump Files Exposed

| Attribute | Value |
|-----------|-------|
| **Severity** | **MEDIUM** |
| **CVSS v3.1** | **6.5** (AV:L/AC:L/PR:N/UI:R/S:C/C:H/I:N/A:N) |
| **Category** | Sensitive Data Exposure |
| **Location** | `/home/pi/n-compasstv-secure/player-server/src/bin/` |

#### Description

Eight core dump files (~330 MB each, ~2.6 GB total) are stored within the vault at `player-server/src/bin/`. Core dumps contain complete memory snapshots of crashed processes, potentially including:

- Decrypted application data
- API keys and credentials in memory
- Database connection strings
- User session tokens
- Private keys and certificates
- Any data the process was handling at crash time

The presence of core dumps indicates the application is crashing regularly, and the dumps have been left in place rather than analyzed and removed.

#### Evidence

```
player-server/src/bin/
├── core (multiple files, ~330 MB each)
└── [8 core dump files, ~2.6 GB total]
```

#### Impact

- Potential exposure of **all in-memory secrets** at time of crash
- Could contain encryption keys, passwords, tokens that were cleared from source code
- Large file size indicates full memory dumps, not truncated
- Regular crashes suggest application stability issues

#### Remediation

1. **Immediate:** Securely delete all core dump files (`shred -vfz core.*`)
2. Configure core dump limits: `ulimit -c 0` or `echo "* hard core 0" >> /etc/security/limits.conf`
3. Disable core dumps system-wide: `kernel.core_pattern=/dev/null` in `/etc/sysctl.conf`
4. Implement proper crash reporting (sentry, crashpad) that sanitizes memory before collection
5. Investigate root cause of crashes — 8 crashes suggests a recurring stability issue

### V-019: AnyDesk Remote Access Service

| Attribute | Value |
|-----------|-------|
| **Severity** | **MEDIUM** |
| **CVSS v3.1** | **6.2** (AV:L/AC:L/PR:N/UI:N/S:U/C:H/I:N/A:N) |
| **Category** | Unnecessary Attack Surface / Remote Access |
| **Location** | `AnydeskService` in player-server services |

#### Description

The application includes an AnyDesk service integration (`AnydeskService`), indicating the digital signage player uses AnyDesk for remote access and management. While AnyDesk is a legitimate remote administration tool, its presence on a production digital signage device:

- Expands the attack surface with a persistent remote access vector
- AnyDesk vulnerabilities could allow unauthorized remote control
- Default or weak AnyDesk passwords are a common misconfiguration
- Remote access tools are high-value targets for persistence

#### Evidence

```
player-server/src/services/AnydeskService.js
```

The service is integrated into the application's service layer, suggesting active use.

#### Impact

- Potential unauthorized remote access to the device
- Persistent access vector that survives reboots
- AnyDesk vulnerabilities (historically CVE-2024-8068, CVE-2024-8069) could be exploited
- Difficult to detect if AnyDesk runs as a background service

#### Remediation

1. **Evaluate necessity** — is AnyDesk required, or can it be replaced with SSH + key-based auth?
2. If required:
   - Change AnyDesk password to a randomly generated 20+ character string
   - Enable unattended access password separately from the GUI password
   - Restrict AnyDesk to specific IP ranges if possible
   - Keep AnyDesk updated (enable auto-update)
   - Enable AnyDesk logging and monitoring
3. If not required:
   - Uninstall AnyDesk completely
   - Remove the `AnydeskService` from the application
   - Audit for any other remote access tools (TeamViewer, VNC, etc.)

### V-020: SQLite Database Unprotected

| Attribute | Value |
|-----------|-------|
| **Severity** | **HIGH** |
| **CVSS v3.1** | **7.5** (AV:L/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:N) |
| **Category** | Sensitive Data Exposure / Missing Encryption |
| **Location** | `/home/pi/n-compasstv-secure/db_backup_dirty/_data.db` and `player-server/src/db/_data.db` |

#### Description

The SQLite database file (`_data.db`) is stored without any encryption. Two copies exist:

1. **Live database** at `player-server/src/db/_data.db` — actively used by the application
2. **Backup copy** at `db_backup_dirty/_data.db` — a backup labeled "dirty" (suggesting unclean shutdown)

Both copies contain the full schema with 18 tables holding operational data, API keys, license keys, device information, content metadata, and usage logs. SQLite's default configuration provides no encryption, access control, or access logging.

#### Evidence

```sql
-- Database file: _data.db (86 KB, 18 tables)
-- Tables: global_settings, license, contents, playlist_contents,
--         host_info, programmatic_ads, vistar_content, computer_usage,
--         internet_logs, template_zones, error_logs, update_logs,
--         filler_logs, version, piclaw_demo_faults, [3 more]
```

The "dirty" backup label suggests the backup was taken during an unclean state, which could also indicate data integrity concerns.

#### Impact

- **Single file exfiltration** grants access to ALL operational data
- No access control — any user/process can read the database
- No encryption — even filesystem-level encryption wouldn't protect against root access
- Backup copy creates additional exposure point
- Database can be modified to inject false data (content, playlists, settings)

#### Remediation

1. **Immediate:** Migrate to SQLCipher (encrypted SQLite) or a client-server database with authentication
2. If SQLite must be used:
   - Enable SQLCipher encryption: `PRAGMA key = '...'` on every connection
   - Set file permissions to 600 (owner read/write only)
   - Restrict database file to application user only
3. **Encrypt at rest:** Ensure the filesystem/vault provides adequate protection (which it currently does not, given V-012)
4. **Remove the backup:** Delete `db_backup_dirty/_data.db` or move to encrypted backup
5. **Add audit logging:** Log all database access attempts and modifications
6. Implement database integrity checks (hash verification) to detect tampering

## 9. Conclusion

This engagement demonstrated that **strong cryptography is rendered useless by weak key management**. The LUKS2 vault using AES-XTS-512 with argon2id KDF represents industry-best encryption — yet the entire vault was compromised in under 15 minutes using only the hardcoded Pi serial number found in a plaintext Python script.

The root cause is a chain of security failures:

1. **The encryption key was derived from publicly available hardware information** (Pi serial number)
2. **The key was hardcoded in source code** rather than generated randomly
3. **The key was stored alongside the vault** on the same filesystem
4. **Supporting data (database, config, credentials) was unencrypted** within the vault
5. **No defense-in-depth measures existed** — single point of failure
