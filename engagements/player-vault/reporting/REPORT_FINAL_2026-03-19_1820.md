## 3. Findings

### V-012 — Hardcoded Vault Passphrase

- **Severity:** Critical
- **CVSS Score:** 9.8
- **Affected Target:** N-Compass TV Vault (vault.img)
- **Description:** The LUKS vault is unlocked by a passphrase embedded in hardware_lock.py and derived from the Raspberry Pi serial number.
- **Evidence:** ```
hardware_lock.py revealed AUTHORIZED_PI_SERIAL = "ffb6d42807368154" and .bash_history contained the exact echo command used for decryption.
```
- **Impact:** Any attacker with local access can decrypt vault.img and access player-server backups, logs, and database snapshots.
- **Remediation:** Move key material off device, prompt for passphrase entry on device console, avoid storing credentials in source code, and rotate passphrase per device.
- **Hardening:** Use TPM or secure enclave to seal keys to hardware, require two-factor confirmation for decryption, and monitor /usr/local/bin for unauthorized modifications.
- **References:** CWE-798, Hardcoded Credentials

### V-013 — Newline-Sensitive Passphrase Flow

- **Severity:** High
- **CVSS Score:** 7.5
- **Affected Target:** Vault unlock script
- **Description:** Vault unlock requires `echo "passphrase" | sudo cryptsetup open ...` and fails if the newline is omitted, revealing fragile automation. Attackers could replicate this flow with minimal tools.
- **Evidence:** ```
Attempts using `echo -n` failed while exact recorded command worked; script uses `echo` with newline when piping into cryptsetup.
```
- **Impact:** A local user or script with sudo privileges can decrypt vault.img instantly; the automation detail lowers the barrier for replication.
- **Remediation:** Use key files or protected key slots instead of echoing passphrases, and remove super-user scripting around sensitive operations.
- **Hardening:** Implement passphrase validation with key files, store secrets in encrypted keystore, and audit sudo scripts for unsafe echo usage.
- **References:** CWE-117, Cryptsetup best practices

### V-014 — Device-Bound Vault Key Exposure

- **Severity:** High
- **CVSS Score:** 7.2
- **Affected Target:** Player source tree
- **Description:** The device unlock key relies solely on the Raspberry Pi serial number which is public, so any cloned player can decrypt other vaults if the scripts remain unchanged.
- **Evidence:** ```
unlock_vault.py and hardware_lock.py derive the passphrase simply from /proc/cpuinfo serial; Serial `ffb6d42807368154` decoded to the passphrase used for recovery.
```
- **Impact:** Possessing a single authorized Pi or a buildable clone allows decrypting any vault produced by the same factory script.
- **Remediation:** Use per-device key material stored securely, incorporate tamper detection, and enforce hardware unique unlocking (TPM / HSM).
- **Hardening:** Add code signing for unlock scripts, enforce encrypted key retrieval from a secure service that binds to device identity, and monitor for unauthorized copies in deployment.
- **References:** Supply chain hardening guidance

---

## 4. Security Enhancement Recommendations

### Secret Management

Store vault passphrases in dedicated KMS/TPM hardware instead of plaintext scripts, and rotate keys using authorized deployments only.

### Local Access Controls

Limit which accounts can execute cryptsetup and remove convenience sudo entries from scripts to prevent misuse by less-privileged users.
